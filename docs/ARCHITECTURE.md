# 프로젝트 공통 관심사 처리 전략

이 프로젝트에서는 각 기능의 목적과 요구되는 데이터의 수준에 따라 Filter, Interceptor, AOP를 구분하여 사용합니다.

## 1. Filter (서블릿 계층)
스프링 컨텍스트 외부 또는 최전방에서 처리해야 하는 로직을 담당합니다.

*   **모든 API 요청 로그 기록**
    *   **이유:** Controller에 도달하기 전의 Raw한 HTTP 요청 데이터(Body, Header 등)를 기록하기 위함입니다.
*   **암호화된 요청 복호화**
    *   **이유:** 스프링이 HTTP Body를 JSON 객체로 변환(Message Conversion)하기 전에 원본 데이터를 복호화해야 정상적인 데이터 바인딩이 가능합니다.

## 2. Interceptor (스프링 MVC 계층)
스프링 빈에 접근 가능하며, 핸들러(Controller) 정보가 필요한 로직을 담당합니다.

*   **특정 등급 유저 접속 허용 (권한 제어)**
    *   **이유:** 유저의 세션 정보뿐만 아니라, 현재 요청이 도달할 Controller의 권한 정보(Annotation 등)를 파악하여 접근 제어를 수행하기 위함입니다.

## 3. AOP (비즈니스 로직 계층)
메서드 단위의 세밀한 제어가 필요한 공통 로직을 담당합니다.

*   **DB 작업 롤백/커밋 제어 (Transaction)**
    *   **이유:** 비즈니스 로직(Service) 단위의 원자성을 보장하기 위해 서비스 메서드 시작과 끝을 트랜잭션으로 감쌉니다.
*   **메서드 실행 시간 측정**
    *   **이유:** 특정 서비스나 컴포넌트의 비즈니스 메서드만 타겟팅하여 성능을 측정하고 병목 지점을 파악하기에 가장 적합합니다.
